# M10 — 负载 harness 与容量曲线（设计）

状态：**已实现**。

对应任务 10.1–10.4（L2 层交付物）。设计方法论来自 D9：**不写死并发数字，
交付一条带拐点的负载/延迟曲线**，并指明拐点先撞哪面墙。压测负载 = 真实
热路径 + mock 模型——真音频帧穿过真传输 / 去重水位 / 内联 VAD / 环形缓冲 /
双背压 / 编排，只有模型段是注入了真实时延形状的 mock，度量的是内核而非
模型吞吐。

## 1. 组成

| 部分 | 位置 | 职责 |
|---|---|---|
| 虚拟会话 worker | `internal/loadgen/worker.go` | 拨号/握手/逐轮对话循环，client 侧首响与打断采样 |
| netem 等效整形器 | `internal/loadgen/shaper.go` | 上行到达时序扰动：延迟/抖动/突发，严格保序 |
| 指标抓取 | `internal/loadgen/scrape.go` | `/metrics` 文本解析、直方图**窗口差分**求分位 |
| ramp 控制器 | `internal/loadgen/loadgen.go` | 阶梯扩容、warmup、逐步采集 |
| 报告与归因 | `internal/loadgen/report.go` + `html.go` | CSV/JSON/自包含 SVG 曲线页、拐点与撞墙分析 |
| 入口 | `cmd/loadgen` | 命令行；服务端配置见 `configs/loadtest.yaml` |

mock 时延经 `adapters.mock` 配置注入（M4 的 `Latency` 早已支持，本里程碑补
配置管道）：ASR final 60ms、LLM 每 token 25±10ms、TTS 每帧 18ms——合成略快
于实时，使每轮响应有真实的"在飞"窗口，可被打断、可被排队拉长。默认值全零，
demo 与既有测试行为不变。

## 2. 关键设计决策

### 2.1 每路负载驱动真实热路径

worker 与浏览器客户端走完全相同的路径：WS 二进制帧、M8 握手（START/ack）、
**连续不间断**的 50fps 上行（说话与静音帧一样发，VAD 边界由服务端判定）、
下行帧解析。一轮 = 说话 600ms → 静音 → 等首个下行音频帧（client 端到端
首响）→ 等下行静默 350ms（轮结束）。每第 4 轮在响应播放中途再次开口，
测从"开口"到收到 `BARGE_IN` 控制帧的端到端打断时延。

### 2.2 两个观测口径，刻意都保留

- **client e2e**：从用户动作的**理想时刻**起算（见 2.3），含 VAD 滤波窗、
  网络、排队——用户实际感受；
- **服务端直方图差分**：每步前后各抓一次 `/metrics`，桶计数相减得到
  **本窗口**的直方图再插值求分位——内核自己的账目（首响从停顿起算、
  打断取消从 cancel 发起起算，即 200ms 预算口径）。

两者的分工在 netem 对照里看得最清楚（§4）。直方图分位受桶分辨率限制
（如 srv p50 恒为 150ms 是 100–200ms 桶内插值），client 样本是精确值。

### 2.3 netem 等效整形器：只造"到达时序扰动"，不伪造乱序

规格明确：WS/TCP 之上链路丢包表现为**重传引发的延迟尖峰与突发交付**，
应用层永远看不到乱序/缺帧。整形器据此只移动帧的发出时刻且**严格保序**——
`release(i) = max(release(i-1), ideal(i) + delay + jitter + burstHold)`，
单调钳制本身就是 TCP 队头阻塞语义：一帧被延迟，其后所有帧跟着排队，
接收侧观察到静默后突发。Linux 下可改用内核级注入获得同等效果：

```
tc qdisc add dev lo root netem delay 40ms 20ms loss 1%
```

本项目开发机为 Windows，进程内整形是跨平台等效实现，且可逐 worker 独立
播种、确定性复现。

测量口径配套：首响 t0 取最后一个说话帧的**理想网格时刻**（用户何时停止
说话），而非整形后的实际发出时刻——否则 netem 的基础延迟会被吸收进 t0，
扰动在 p50 上不可见（实现初版正是如此，已修正）。

### 2.4 相位锁定与入场摊开

mock 驱动的轮持续时间几乎相同，同一时刻入场的 worker 会**永久同步**，
以波峰形态打服务器——波峰需求远高于稳态均值，曲线测的就不再是稳态容量。
对策：每步新 worker 的入场随机摊开在整个 warmup 窗口内；阶梯渐进扩容
（每步只新增部分会话）让相位随历史自然散开。首版曾以 500 路一秒内风暴
入场起步，拐点位置被波峰干扰严重——这是 harness 自身的测量学课题，
记录在案。

### 2.5 拐点判定：持续性规则

单个降级窗口（GC、操作系统毛刺、后台进程）不是容量拐点——**容量耗尽
不会因为加更多负载而自愈**。故 `Analyze` 要求降级信号（p99 > 2×基线+20ms、
丢帧率 > 1%、出现错误）**延续到下一更重的步**（或已是最后一步）才判拐点。
首版在 2→320 阶梯中把并发 40 处的孤立瞬态误判为拐点，由此引入该规则并
固化为单测。

### 2.6 超时重对齐

过拐点后部分轮超时（这是测量值不是 harness 缺陷）。超时后 worker 先吞掉
迟到的残余响应再开下一轮，否则上一轮的迟到音频会被误认作新轮首响，产出
虚假的快样本——首版 1000+ 并发段的 p50 反而"变好"即此伪象，已修正。
过拐点区域的 client 样本仍以近似看待，服务端直方图为权威。

## 3. 实测容量曲线（10.3 / 10.4）

环境：i7-13620H（16 逻辑核）/ Windows 11 / **安静模式电源方案（功率受限）**；
服务端与 loadgen 同机（曲线是该机器该状态下的**容量下界**）。阶梯
50→1500，每步 10s，打断每 4 轮一次。工件：`docs/load/capacity.{csv,json,html}`。

| 并发 | 轮 | 错误 | e2e p50 | e2e p99 | srv首响p99 | 内核开销p99 | 内核取消p99 | 丢帧率 | CPU | turns/s |
|---|---|---|---|---|---|---|---|---|---|---|
| 50 | 162 | 0 | 416ms | 438ms | 199ms | 5ms | 1ms | 0 | 6% | 21 |
| 100 | 339 | 0 | 414ms | 415ms | 199ms | 5ms | – | 0 | 0.5% | 44 |
| 200 | 676 | 0 | 414ms | 416ms | 199ms | 5ms | – | 0 | 0.6% | 86 |
| 300 | 985 | 0 | 414ms | 418ms | 199ms | 5ms | – | 0 | 1.6% | 130 |
| **400** | 1264 | 0 | 422ms | 590ms | 329ms | 72ms | 45ms | 0 | **42%** | 165 |
| **500** | 1585 | 0 | 511ms | 1377ms | 445ms | 103ms | 143ms | 0.02% | **75%** | **202** |
| 750 | 1508 | 6 | 590ms | 3933ms | 480ms | 229ms | 91ms | **21.5%** | 80% | 199 |
| 1000 | 1116 | 233 | 605ms | 5181ms | 757ms | 436ms | 96ms | 37.3% | 80% | 161 |
| 1500 | 1046 | 133 | 530ms | 7020ms | 480ms | 142ms | 25ms | 38.4% | 78% | 139 |

读法：

- **平台区（50–300）**：一切平直。e2e p50 414ms 可逐项对账 = hangover
  300ms + ASR final 60ms + LLM 首 token ~35ms + TTS 首帧 ~19ms；内核开销
  p99 仅 **5ms**——首响几乎全是模型固有时延与配置的 VAD 等待，内核自身
  开销可忽略（M5 分解口径的实测验证）。
- **拐点（~400–500）**：CPU 从 2% 跳 42% 再 75%，p99 抬头，吞吐在 500 处
  达峰值 202 轮/s。**先撞的墙是 CPU**（在安静模式功率上限下表现为 ~80%
  即饱和；mock 定时器 + 帧编解码 + 调度是主要消耗，且含同机 loadgen 争用，
  harness 机械判定保守地标注为 scheduling）。
- **过墙（750+）**：丢帧率从 0 跳 21.5%→38%——**双背压按设计在入口卸载**
  （`egress_drop` 全程为 0：压力沿 TTS→token→finals→ASR 链传回，最终
  表现为入口 drop-oldest）。轮速率缓降不崩塌（139–199/s），srv 首响 p50
  仍 ~160ms（完成的轮依然快，损害集中在尾部）；进程存活、heap 有界
  （34–171MB 波动，无单调增长）。
- **打断预算**：内核取消 p99 在全部负载档位 ≤143ms，**200ms 预算在三倍
  拐点负载下仍成立**——被打断意味着子链取消，而取消路径不经过拥塞的
  egress 队列，这是打断优先架构的直接收益。e2e 打断 p99 过墙后劣化
  （VAD 判定本身要排队），口径分开后两件事互不混淆。
- **恢复**：另一次压到 5000 并发（丢帧 22–38%、超时上千）后，60s idle
  回收完成，服务端回落至 **0 会话 / 7 goroutines**——过载不留疤（8.4 的
  无泄漏性质在 30 倍于其测试规模下复验）。

## 4. netem 扰动对照（10.2 验收）

固定 100 并发（平台区），注入 delay 40ms + jitter 30ms + 每秒 120ms 突发持留：

| 口径 | clean | netem | Δ |
|---|---|---|---|
| e2e 首响 p50 | 417ms | 475ms | **+58ms**（基础延迟+抖动均值如实计入） |
| e2e 首响 p99 | 438ms | 552ms | **+114ms**（突发持留的尾部） |
| e2e 打断 p99 | 91ms | 160ms | +69ms |
| srv 首响 p99 | 199ms | 199ms | **0** |
| 丢帧 / 错误 | 0 / 0 | 0 / 0 | 突发被入口侧消化 |

服务端口径不动是**正确的**：内核从自己听到的 VAD 边界（到达域）起算，
上行扰动平移轮的起点而不进内核账目。网络损伤落在 e2e、不落在内核——
这正是"内核开销与环境时延分开度量"的诚实边界在扰动下的版本。

## 5. 诚实边界（如实记录）

- **同机争用**：loadgen 与服务端共享 CPU，曲线是容量**下界**；拐点附近
  的 scheduling 归因含 loadgen 自身的份额。分机压测会推后拐点。
- **机器状态敏感**：安静模式（功率受限）下拐点 ~500；同机前夜不同电源
  状态下 320 并发仅 1% CPU。这正是 D9 不交付单一数字的原因——交付的是
  方法、harness 与曲线形状；绝对数值随硬件/电源状态浮动，复现命令在
  报告头部。
- **CPU 利用率为 runtime 估计**（`runtime/metrics` busy/capacity），趋势
  分析够用，不是记账值。
- **fd 墙未触及**：每会话 1 socket，本曲线最高 1500 远低于 Windows 动态
  端口（~16k）与句柄上限；在更高并发到达 fd 墙之前 CPU 墙先到。
- **过拐点的 client 样本是近似**（超时/重对齐残余），服务端直方图为权威。

## 6. 验收 / 测试

全部 `-race` 干净：

- 整形器：基础延迟精确、抖动有界、**保序不变量**（千帧扫描）、突发窗口
  持留后整批释放、下一窗口重复。
- 分位算术：client 样本最近秩分位；直方图差分 + 桶内插值（+Inf 桶钳制到
  最高有限界，绝不外推）。
- 拐点分析：无拐点 / CPU 墙 / 丢帧墙 / 错误优先 / **瞬态毛刺不算拐点**
  各一夹具。
- 抓取：文本格式解析、非 200 拒绝。
- e2e：进程内起完整真实栈（transport+session+VAD+pipeline+延迟 mock），
  2 步 ramp、5ms 帧间隔（快于实时，PTS 仍按 20ms 名义推进——内核走采样
  时钟，全链路与实时等价）、netem 开启、断言逐步有轮/有首响样本/打断
  落地/服务端直方图有数/零错误。
