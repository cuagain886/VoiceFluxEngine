# M3 — SPSC 无锁环形缓冲（设计）

状态：**已实现**。

对应规格`ring-buffer`和设计决策**D2**（仅在音频边界处环）、**D3**（双背压）。任务：3.1–3.5。

## 1. 目的与放置

环形缓冲是音频热路径原语——它**恰好存在两处**（D2）：

```
传输 ──入口环──▶ ASR          (一个生产者：WS读goroutine)
TTS  ──出口环──▶ 传输         (一个生产者：TTS阶段)
```

文本阶段之间的一切（ASR→LLM、LLM→TTS）使用普通channel：文本是低频且*阻塞那里就是背压机制*(D3)。音频是高频（50fps）且绝不能阻塞套接字读者，所以环吸收突发并通过丢弃**最陈旧**帧来减轻负载——对于实时语音，新帧比完整历史更值得。

入口严格保持SPSC因为VAD在读goroutine中*内联*运行(D4在`design.md`)，不是第二个消费者。

## 2. 核心设计问题：drop-oldest 打破经典 SPSC

教科书SPSC环仅需要两个原子游标——生产者写`tail`，消费者写`head`——完全无CAS。我们故意**没有**这样构建，因为规格要求`drop-oldest`策略：当环满时生产者必须驱逐最旧元素并保留最新。

驱逐意味着**生产者推进`head`**——读游标突然有两个写者。用普通游标这是无法修复的：

- 如果生产者只是覆写槽位，消费者可能正在复制那个确切的槽位 → 撕裂读，一个硬数据竞争在`-race`下。
- "读，然后验证和丢弃"变体仍执行竞争读；竞争检测器（正确地）标记它们即使值被丢弃。

### 解决方案：每槽序列门(Vyukov bounded-queue方案)

每个槽携带自己的`atomic.Uint64`序列，编码槽相对单调增长游标的状态：

```
seq == t        槽空闲；位置t的生产者可写它
seq == t + 1    槽已发布；位置t的消费者可读它
seq == h + cap  在位置h消费的槽；下一圈再空闲
```

- **推**(仅生产者，无CAS)：检查`slots[t&mask].seq == t`，写值，释放`seq = t+1`，推进`tail`。
- **弹**(消费者*和*生产者驱逐时 ⇒ **对`head`的CAS**)：检查`seq == h+1`，然后`CAS(head, h, h+1)`。谁赢CAS赢得槽直到以`seq = h+cap`释放。失败者重载并重试。
- **丢弃最旧**然后就是平凡安全：`推`满时 = `弹()`(丢弃，`dropped++`) + 重试`推`。

每个对`slot.val`的数据访问通过槽自己的原子排序，所以结构*按构造*无竞争——`-race`清洁是算法的属性，不是幸运调度。

一个微妙性这方案也修复：持着陈旧`head`的消费者能通过有符号序列差分分辨"空"(`seq < h+1`)和"head在驱逐下移到我下方"(`seq > h+1`)，重试而非错误报告空。

**权衡成本**：每弹一个CAS（无争用在稳态中——生产者仅在满时触及`head`），每槽8字节序列。**我们保留**：推无CAS、FIFO顺序、有界内存、可证明无竞争驱逐路径。

### 已知限制(已接受)

在驱逐期间生产者可能短暂旋转而消费者在~2指令声明窗口内(CAS赢，`seq`未发布)。结构是无锁的，非等待。窗口是几纳秒；如果消费者goroutine恰好在那里被取消调度，生产者旋转直到它返回。对v1可接受；为诚实性注记(CLAUDE.md约定4)。

## 3. 内存布局 — 游标在私有缓存行上

```go
type Ring[T any] struct {
    mask   uint64        // 后New后只读
    policy Policy        //   "
    slots  []slot[T]     //   "
    _      [64]byte
    tail    atomic.Uint64 // 每次推写~(生产者核)
    _      [64]byte
    head    atomic.Uint64 // 每次弹写~(消费者核)
    _      [64]byte
    dropped atomic.Uint64
    _      [64]byte
}
```

`tail`和`head`由*不同核*以帧速率写。如果它们共享缓存行，一个核上的每次写会使另一个核上的行失效(假共享)，双方都会在缓存一致性流量上停滞。前导填充也保持热游标离只读头字段(`mask`、`slots`)两边都会在每操作加载。

在13代i7(16线程)上测得，相同算法有和无填充，`int`元素，容量1024：

| 变体 | ns/op(稳态) | allocs/op |
|---|---|---|
| `BenchmarkSPSCPadded` | **~55** | 0 |
| `BenchmarkSPSCUnpadded` | ~320–340 | 0 |

**从192字节填充得到≈6×吞吐。** 这是"为什么填充"的量化答案(任务3.5)；无填充孪生在`ring_bench_test.go`中生活且仅测试。

相邻*槽*按设计仍共享缓存行——填充每个槽会乘以内存脚印并摧毁顺序排液的缓存局部性。热争用在游标上，填充去那里。

## 4. 稳态零分配

两半：

1. **环本身**：槽在`New`中预分配；`推`/`弹`复制元素值入/出槽。无每操作分配，永不。
2. **负载字节**(`BufferPool`)：固定容量`[]byte`的空闲列表，实现为*另一个*`环[[]byte]`在数据环相反方向流——数据生产者是池的唯一`Get`者，数据消费者其唯一`Put`者，所以SPSC纪律由交换角色保留。`sync.Pool`被拒绝：`Put(&b)`在每周期上堆分配切片头，打败目的。

池降级而非失败：在耗尽的空闲列表上`Get`回退到`make`(在`Misses()`中计数——M9的指标钩子)，`Put`超额或不适尺寸缓冲让它们落在GC。

门(CI中通过`go test` + bench)：

- `TestSteadyStateZeroAlloc`：在完整池→环→池周期上用640字节PCM帧的`testing.AllocsPerRun` == **0**。
- `BenchmarkSPSCAudioFrame`(现实入口形状，640字节帧通过池+环)：**~175 ns/op, 0 allocs/op**。

## 5. 全缓冲策略(D3 双背压)

| 策略 | 行为 | 使用在 |
|---|---|---|
| `DropOldest` | 驱逐最陈旧元素，`Dropped()`++，保留最新 | 音频入口/出口(默认) |
| `Reject` | `推`返回`false` → 调用者感觉背压 | 池空闲列表；任何无损地方 |

可配置每边通过`ring_buffer.ingress_policy` / `egress_policy`(通过`ringbuf.ParsePolicy`在配置加载时验证)。丢弃计数器是M9丢弃率指标和M10"我们首先打到哪面墙"分析的原始输入。

确定语义检查(`TestDropOldestDeterministic`)：推0..11进容量8 DropOldest环产生`Dropped()==4`和弹4..11——最旧被驱逐，最新被保留，顺序保留。

## 6. 其他决策

- **游标是`uint64`且永不处理换回。** 在50fps会话需要~11.7亿年溢出。有符号差分比较(`int64(seq - (h+1))`)无论如何换回正确。
- **`Len()`是一个竞争快照**(两个独立原子加载)，夹住在`[0, Cap]`——对指标好，记录为这样，从不用于正确性。
- **泛型`环[T]`**：管道(M5)将用其帧结构实例化它；池实例化`环[[]byte]`。无接口装箱——元素内联存储在槽中，这也是使零分配属性成立的原因。
- **弹出槽被清零**(`s.val = zero`)所以`[]byte`承载元素不会钉住其支撑数组直到槽下一圈。

## 7. 验收 / 测试(3.5)

- `-race`套件：并发下无损FIFO顺序(`TestConcurrentFIFO`，100k元素，确切序列)、并发驱逐与记帐`收到 + 丢弃 == 推送`且严格单调顺序(`TestConcurrentDropOldest`，容量8强制常数驱逐竞争)。
- 空弹返回`ok=false`无阻塞；满时拒绝离开丢弃计数器未触及；非2的幂容量在`New`处拒绝。
- 基准：见上表；`BenchmarkDropOldestSaturated`(生产者消费者完全停滞) ≈ 100 ns/op, 0 allocs——最坏情况仍花约两个正常操作(一个驱逐+一个推送)，无悬崖。
