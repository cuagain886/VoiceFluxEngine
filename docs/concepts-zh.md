## §1 音频：声音在计算机里长什么样

这一节是全项目的地基——不懂音频，后面的"帧""时序""丢帧"都是空中楼阁。

### 声音 / 声波
声音本质是空气压强随时间的连续起伏（波）。麦克风做的事就是：每隔一小段
时间量一次当前的空气压强，记成一个数。

### 采样（Sampling）/ 采样率（Sample Rate）
- **定义**：把连续的声波，每隔固定时间量一次、变成一串离散的数，这个"量"
  的动作叫采样；每秒量多少次叫采样率，单位 Hz。
- **类比**：像拍延时摄影——连续的日出，你每秒拍一张，拍得越密越接近真实。
- **本项目**：采样率 **16000 Hz（16kHz）**，即每秒量 16000 次。人声主要
  能量在 8kHz 以下，按采样定理 16kHz 足够还原人声（电话就是 8kHz，16kHz
  已经清楚）。音乐用 44.1kHz，但语音不需要那么高——**用最低够用的采样率
  省带宽和算力**，这是本项目"假设 16k/mono"的由来。

### 位深（Bit Depth）/ 16-bit
- **定义**：每个采样点用多少位（bit）来存。16-bit = 每个点是一个 16 位
  整数（范围 -32768 ~ 32767）。决定声音振幅的精度，也就是动态范围和量化噪声水平
- **类比**：采样率是"多久量一次"，位深是"每次量得多精细"。16 位 = 量出
  的数有 65536 个档位，对人声足够。
- **本项目**：16-bit ⇒ **每个采样点占 2 字节**。这就是后面"640 字节"算
  出来的关键。

### 声道（Channel）/ 单声道（Mono）
- **定义**：几路独立的声音。单声道（mono）= 1 路；立体声 = 2 路（左右耳）。
- **本项目**：**单声道**。语音对话不需要立体声，1 路省一半数据，也让
  "一个采样点 = 一个数"的模型最简单。

### PCM（脉冲编码调制 / Pulse Code Modulation）
- **定义**：上面这一整套"采样+量化"得到的**原始数字音频格式**——就是一长
  串采样点整数，没有任何压缩。
- **类比**：PCM 之于音频 ≈ BMP（位图）之于图像——最原始、最大、但最简单
  直接，拿来就能算。MP3 之于 PCM ≈ JPG 之于 BMP（压缩格式，要解码）。
- **本项目**：全程假设线路上就是裸 PCM（16k/16bit/mono），**不做编解码**
  （FFmpeg 转码被推迟到未来）。这是个刻意的范围收窄：先把时序/背压/打断
  做对，编解码是另一码事。

### 帧（Frame）
- **定义**：把连续的 PCM 流切成固定时长的小段，每段叫一帧。
- **类比**：视频是一帧帧图像；音频帧是"一小段声音"。
- **本项目**：**20ms 一帧**。算一下这一帧多大：
  ```
  16000 采样点/秒 × 0.02 秒 = 320 个采样点
  320 个采样点 × 2 字节/点（16-bit）= 640 字节
  ```
  所以"640 字节 = 一帧 = 20ms 音频"是贯穿全项目的基本单位。每秒
  1000ms ÷ 20ms = **50 帧/秒（50fps）**——这就是 deep-dive 里反复说的
  "音频两端高频 50fps"的来历。
- **为什么 20ms**：太大则延迟高（要攒够 20ms 才发一帧），太小则开销大
  （每帧都有头部和系统调用）。20ms 是实时语音的常见折中。

### PTS（呈现时间戳 / Presentation Timestamp）
- **定义**：每帧音频"应该在第几微秒播放"的时间标签。
- **类比**：电影胶片每一格上标着"第几秒放"，放映机照着它对齐音画。
- **本项目**：帧头里的 `ts_us`（微秒）。它**不是用墙上的钟算的**，而是用
  "已经过了多少个采样点 ÷ 采样率"算的（采样时钟）——这样不会因为程序卡顿
  累积漂移。下行音频靠 PTS 让浏览器平稳播放，字幕 token 靠它对齐到对应的
  语音时刻。

### RMS / 能量（Energy）
- **定义**：一帧里所有采样点的"均方根"，衡量这一帧有多响。安静≈0，说话时大。
- **类比**：音量表上下跳的那根条，就是能量的可视化。
- **本项目**：VAD（见 §3）就靠每帧的 RMS 能量判断"现在有没有人在说话"——
  能量超过门限 = 有人声。计算极便宜（一帧几微秒），所以能内联在热路径里。

### AEC（声学回声消除 / Acoustic Echo Cancellation）
- **定义**：扬声器放出的 Agent 声音会被麦克风重新录进去，AEC 负责把这部分
  "自己的回声"减掉，避免系统把自己的声音当成用户说话。
- **类比**：开免提打电话时不会听到自己声音的回声，就是 AEC 在工作。
- **本项目**：**依赖浏览器内置 AEC**（`getUserMedia({echoCancellation:true})`），
  内核不自己做。这很关键——没有 AEC，Agent 一出声就会把自己的声音当成用户
  打断，打断功能直接失灵。这也是误触发抑制的第一道防线。

## §2 三个模型：ASR / LLM / TTS（本项目把它们当"租户"，不自己实现）

一轮语音对话的流水线是：**你说话 → ASR 转文字 → LLM 想回复 → TTS 合成
语音 → 播放给你**。这三个就是其中的 AI 模型。本项目不训练也不实现它们，
只定义统一接口、用 mock 假货跑通，并早早接了一个真实 LLM 验证接口形状。

### ASR（自动语音识别 / Automatic Speech Recognition）
- **定义**：把语音音频转成文字。也叫 STT（Speech-to-Text）。
- **类比**：手机语音输入法、会议自动字幕。
- **本项目**：输入音频帧流，输出文字。注意两种输出：
  - **partial（部分/中间结果）**：边听边猜，"你好…""你好世界…"会随着你
    继续说而不断刷新——这让字幕能实时跟着你的嘴动。
  - **final（最终结果）**：一句话说完了，给出确定的最终文字。
  一次 ASR 流 = 一句话（一个 utterance）。

### LLM（大语言模型 / Large Language Model）
- **定义**：理解文字并生成回复的 AI，比如 GPT、DeepSeek、Claude。
- **本项目**：输入用户的话（+ 历史），**流式**吐出回复。
  - **token（词元）**：LLM 不是一次吐出整句，而是一小块一小块地吐
    （一个 token 大约是一个词或几个字）。"流式"就是指边生成边吐 token，
    不用等整句生成完——这是实时对话感的来源。
  - 本项目在 M4 就**早早接了一个真实云 LLM**（OpenAI 兼容的 SSE 接口，
    默认 DeepSeek），专门验证"我设计的统一接口，真实模型塞得进去吗"。

### TTS（文字转语音 / Text-to-Speech）
- **定义**：把文字合成成语音音频。ASR 的逆操作。
- **类比**：导航软件念路名、有声书朗读。
- **本项目**：输入 token 流，输出音频帧流（又回到 §1 的 PCM 帧）。mock 版
  合成的是"静音"（按时长产出零值 PCM），因为本项目验证的是时序而非音质——
  能听到节奏对不对就够了，真实 TTS 是可插拔的另一码事。

### 流式 vs 阻塞（Streaming vs Blocking）—— 本项目的灵魂
- **阻塞式**（市面多数简单实现）：录完整句 → 等 ASR 出完整结果 → 等 LLM
  出完整回复 → 等 TTS 合成完 → 才开始播。用户体验：3~5 秒延迟，中途没法插话。
- **流式**（本项目）：三个阶段**并发重叠**——LLM 吐第一个 token，TTS 就
  开始合成它，同时 LLM 继续吐下一个。用户体验：几百毫秒就开始听到回应，
  且能随时打断。`dash.html` 的瀑布图画的就是这种"重叠"省了多少时间。

## §3 语音交互：VAD 与"打断"

这是本项目三大核心之一（时序/背压/**打断**），也是最体现"对话感"的部分。

### VAD（语音活动检测 / Voice Activity Detection）
- **定义**：实时判断"现在这一帧音频里，有没有人在说话"。
- **类比**：对讲机的声控开关——有声音才传输，安静就不传。
- **本项目**：用最简单的**能量 VAD**——看每帧 RMS 能量（§1）是否过门限。
  它产出两个事件：`speech_start`（开始说话）和 `speech_end`（说完了）。
  这两个事件驱动整个打断逻辑。

### 双门限 / 滞回（Hysteresis / Dual Threshold）
- **问题**：如果只用一个门限，能量在门限上下抖动时会疯狂误判"说话/没说话/
  说话/没说话"。
- **方案**：用两个门限——**高门限**才算"开始说话"，跌破**低门限**才算
  "停止说话"，中间是"维持当前状态"的缓冲带。
- **类比**：空调温控——26℃ 才开制冷，降到 24℃ 才关，不会在 25℃ 反复开关。
- **本项目**：`EnergyThreshold`（进入）和 `ExitThreshold`（退出），中间的
  间隙就是滞回带，抑制抖动误触发。

### 最小语音时长（Min-Speech）
- **定义**：能量超门限必须**持续够久**（如 100ms）才算真的开始说话。
- **作用**：滤掉咳嗽、敲键盘、桌子碰一下这种瞬间噪声。
- **本项目**：默认 100ms。这 100ms 是**防误触发的代价，不算内核开销**——
  所以打断延迟的口径里它被单独计量，不和"内核取消耗时"混在一起（诚实计量）。

### Hangover（语音尾挂 / 静音容忍）
- **定义**：检测到"停止说话"后，再多等一小段（如 300ms）才真的判定一句话
  结束。
- **作用**：人说话中间会有短暂停顿（"我想想…要不…"），没有 hangover 会
  把一句话切成好几段。
- **类比**：打字时的"输入法候选框不会因为你停顿半秒就消失"。
- **本项目**：默认 300ms，要求 ≥ min-speech（配置校验里有这条）。

### Barge-in（打断）—— 北极星指标
- **定义**：Agent 正在说话时，用户开口插话，系统立即停止 Agent 的输出、
  转去听用户的新话。
- **类比**：真人对话里你打断对方，对方会立刻停下来听你说——这才自然。
- **本项目**：从"用户开口"到"Agent 停声"要 **< 200ms**，这是全项目的头号
  指标。实现是：VAD 在 Agent 说话状态下检测到 speech_start → 取消正在进行
  的 LLM→TTS 子链 → 清空还没播的音频 → 通知浏览器停播。关键设计：取消走
  **控制面**不排在数据队列里，所以即使系统过载、下行音频已经堵了几秒，
  打断照样几十毫秒内完成。

### 状态机（State Machine）
- **定义**：系统在任一时刻只处于有限几个"状态"之一，事件触发状态间迁移，
  非法迁移被拒绝。
- **类比**：红绿灯——只有红/黄/绿三态，按固定规则切换，不会"既红又绿"。
- **本项目**：四态——`LISTENING`（听着）/ `SPEAKING_USER`（用户在说）/
  `THINKING`（在想，即 LLM 处理中）/ `RESPONDING_AGENT`（Agent 在说）。
  打断就是 `RESPONDING_AGENT` 状态下收到 `speech_start` 这一个格子的动作。
  用状态机而非一堆 if-else，是因为打断的边角情况（想到一半被打断、旧回复
  迟到）必须每个都显式处理，漏一个就报"非法迁移"。

## §4 网络传输：WebSocket / TCP / 帧协议 / protobuf / SSE

浏览器和服务器之间怎么把音频和文字传来传去。

### TCP
- **定义**：互联网最基础的可靠传输协议——保证数据**按发送顺序、不丢、不重**
  地到达（丢了会自动重传）。
- **本项目要记住的一点**：单条 TCP 连接上，应用层**永远不会看到乱序**。
  这直接决定了本项目"不做会话内重排窗口"（做了是给不存在的问题写代码）。
  TCP 上的"网络抖动"表现为**延迟尖峰和突发到达**（重传导致），不是乱序。

### WebSocket（WS）
- **定义**：在 TCP 之上、让浏览器和服务器能**双向实时收发消息**的协议。
  普通 HTTP 是"请求-响应"一问一答，WebSocket 建立一次连接后两边随时能推
  消息。
- **类比**：HTTP 像写信（一来一回），WebSocket 像打电话（接通后随时说话）。
- **本项目**：用 WebSocket 的**二进制消息**承载音频帧和控制消息。选它而非
  WebRTC，是因为本内核定位在"机房腿"（业界 Twilio、OpenAI Realtime 都用
  WS），且 WebSocket 让"手写流式协议"成为可能（WebRTC 会把分帧/缓冲全
  黑盒掉）。

### WebRTC（对照，本项目没用）
- **定义**：浏览器实时音视频的重型标准方案，自带回声消除、抖动缓冲、拥塞
  控制、NAT 穿透等一整套。
- **为什么本项目不用**：它把所有底层细节黑盒化——用了它，本项目想展示的
  "我自己造的流式协议"就没得展示了；且它的复杂度（SDP/ICE/DTLS/TURN）会
  吞掉本该投入核心难点的时间。它适合"最后一公里"的烂网，本项目定位在它
  之后的干净机房腿。

### 帧协议 / 二进制帧（Binary Frame Protocol）
- **定义**：自己规定的"一条消息的字节怎么排"的格式。本项目每条消息 =
  **24 字节定长头 + 负载**：
  ```
  magic(2) | version(1) | type(1) | seq(8) | ts_us(8) | length(4) | payload...
  魔数       版本         类型       序列号    时间戳      负载长度
  ```
- **类比**：寄快递的标准面单——固定位置写着寄件人、收件人、重量，扫一眼
  就知道怎么处理，不用拆开包裹。
- **本项目**：定长头**解析零分配、与传输无关**（将来换字节流传输也能用，
  因为头里自带 length 能自己分帧）。`magic`（"VS"两字节）是防错的校验，
  version 防版本不匹配，seq 是序列号（去重用），ts_us 是 PTS。

### Payload 编码：裸 PCM vs protobuf
- **本项目的混合策略**：
  - 音频负载 = **裸 PCM 字节**（不套任何编码）——50fps 高频路径上，套
    编码纯属浪费，PCM 本身就是最终格式。
  - 文本/控制负载 = **protobuf**——低频、且字段会演化。

### protobuf（Protocol Buffers）
- **定义**：Google 的一种**结构化数据序列化格式**——你用 `.proto` 文件
  定义数据结构（有哪些字段），工具生成代码，把结构体高效编码成字节、再
  解回来。
- **类比**：JSON 的二进制紧凑版——更小更快，但要先定义 schema。
- **本项目**：用来编码文本消息（带 final/source 标志）和控制消息（START/
  STOP/BARGE_IN/ERROR + 会话 id/epoch/序列号）。好处：M8 给控制消息加三个
  新字段时**零迁移成本**（protobuf 向后兼容）。

### SSE（服务器推送事件 / Server-Sent Events）
- **定义**：一种让服务器**单向、持续地推消息给浏览器**的简单技术（基于
  普通 HTTP）。
- **类比**：订阅了一个频道，服务器有新消息就推给你，你不用反复问。
- **本项目**：仪表盘 `dash.html` 用 SSE 订阅 `/debug/turns`——每完成一轮
  对话，服务器就推一条记录过来，前端画一条新瀑布。为什么不用 WebSocket？
  因为这里只需要服务器→浏览器单向推，SSE 更轻。

### 心跳 / Ping-Pong（Heartbeat）
- **定义**：连接空闲时，定期发个"还在吗"的小包，对方回"在"。久不回 =
  连接已死，及时清理。
- **本项目**：写 goroutine 定期 Ping，超时则判定 attachment 死亡，触发
  重连/回收逻辑。

## §5 并发与数据结构：本项目最硬核的部分

这部分是 Go 工程的精髓，也是面试最容易深挖的地方。

### 并发（Concurrency）/ Goroutine
- **定义**：让多段代码"同时"推进。Go 里用 `goroutine`——极轻量的执行单元，
  一个程序可以开成千上万个。
- **类比**：餐厅里多个服务员同时干活，而不是一个人干完一桌再下一桌。
- **本项目**：每个处理阶段（ASR/LLM/TTS/搬运）各跑在自己的 goroutine 里，
  这样它们才能并发重叠（§2 的流式）。**谁创建谁、谁负责关闭**是核心纪律
  （见 study-guide 的"迷路点速查"）。

### Channel（通道）
- **定义**：Go 里 goroutine 之间**安全传递数据**的管道。一端发，一端收。
- **类比**：流水线上两个工位之间的传送带。
- **本项目**：阶段之间用 channel 传文字（ASR→LLM 的 finals、LLM→TTS 的
  token）。**有界 channel（带容量上限）满了会阻塞发送方**——这正是文本侧
  的背压机制（见下）。

### Context / 取消（Cancellation）
- **定义**：Go 里传递"取消信号"的机制——父任务取消，所有子任务连带取消。
- **类比**：项目经理喊停，所有正在干这个项目的人都立刻收手。
- **本项目**：打断 = 取消"当前轮"的 context，LLM 和 TTS 的 goroutine 收到
  信号立即停止。每次 channel 发送都被 context 守卫，所以打断时**永不挂起**。

### 环形缓冲区 / Ring Buffer（循环缓冲）
- **定义**：一块**固定大小**的数组，读写指针到末尾就绕回开头，循环利用——
  实现一个容量固定的队列，不用动态扩容。
- **类比**：一个有 N 个格子的圆桌转盘，放满了再放就转回第一格覆盖掉最旧的。
- **本项目**：音频两端（入口、出口）各一个环形缓冲。为什么是环不是 channel？
  ①容量固定 ⇒ 内存有界；②支持"满了丢最旧"策略（见 drop-oldest）；
  ③无锁实现下比 channel 快 2.1~3.7×（实测）。

### SPSC（单生产者单消费者 / Single Producer Single Consumer）
- **定义**：这个队列只有**一个**goroutine 往里写、只有**一个**goroutine
  往外读。
- **为什么重要**：SPSC 是最容易做对、做快的并发场景——只有两个角色，不用
  防多个写者互相打架。本项目的入口环严格 SPSC（这就是为什么 VAD 要内联在
  读 goroutine 里——独立成 goroutine 会变成两个读者，破坏 SPSC）。

### 无锁（Lock-Free）/ 原子操作（Atomic）/ CAS
- **锁（Mutex）**：传统并发保护手段——同一时刻只让一个 goroutine 进入临界
  区，其他人排队等。缺点：等待、可能优先级反转。
- **原子操作**：CPU 提供的"不可分割"的操作（读、写、加），不会被其他线程
  打断到一半。
- **CAS（比较并交换 / Compare-And-Swap）**：原子操作的一种——"如果这个值
  还是我预期的旧值，就把它换成新值；否则告诉我失败"。是无锁数据结构的基石。
- **类比**：CAS 像抢座位——"如果这个座位还空着（预期），我就坐下；如果已
  经被人坐了，我重试别的"。全程不用喊"大家都别动我要坐了"（那是锁）。
- **本项目**：环形缓冲全程用原子操作 + CAS，不用锁。drop-oldest 时生产者
  用 CAS 推进读指针来"驱逐"最旧帧——这是全项目最精妙的 15 行（见 M3 文档）。

### 伪共享 / Cache Line（False Sharing / 缓存行填充）
- **定义**：CPU 缓存以"行"（通常 64 字节）为单位搬运。如果两个 goroutine
  频繁修改的两个变量恰好落在**同一缓存行**里，CPU 会以为它们相关，反复
  在核间同步整行——明明不相干却互相拖慢，这叫伪共享。
- **解决**：在变量之间填充无用字节，把它们**撑到不同的缓存行**。
- **本项目**：读指针、写指针、丢帧计数器之间各垫 64 字节填充。**实测：
  去掉填充，吞吐从 33.3ns 掉到 76.1ns（慢 2.3 倍，甚至比 channel 还慢）**。
  这是"伪共享填充不是装饰"的铁证。

### 背压（Backpressure）—— 三大核心之二
- **定义**：下游处理不过来时，**反向给上游施加"慢下来"的压力**，防止数据
  无限堆积、内存爆炸。
- **类比**：生产线下游堵了，传送带停下来，让上游别再往上放——而不是货物
  堆到塌掉。
- **本项目的双背压**（按数据语义分两种）：
  - **音频边缘**：环满了 → **drop-oldest（丢最旧帧）**，绝不阻塞传输——
    实时音频丢旧帧是可接受的（旧音频没价值）。
  - **文本中段**：channel 满了 → **阻塞上游**，绝不丢——丢一个 token =
    回复永久损坏。
    两者串成一条链：慢 TTS → token 满 → LLM 停 → finals 满 → ASR 停 →
    停止抽干入口环 → 最终表现为入口丢帧。**内存有界是这样"构造"出来的，
    不是测出来的**——每个容器都有上限，唯一的泄压口是丢最旧音频。

### drop-oldest vs reject（两种满缓冲策略）
- **drop-oldest**：满了就丢掉最旧的，腾位给新的。用于音频（要最新的）。
- **reject**：满了就拒绝新的（让上游知道、自己处理）。用于需要背压反馈的
  场景。
- **本项目**：可配置，音频边缘默认 drop-oldest。

### 吞吐 vs 延迟（Throughput vs Latency）
- **吞吐**：单位时间能处理多少（如帧/秒、轮/秒）——"流量大小"。
- **延迟**：单个请求从开始到完成花多久（如首响 ms）——"等待时间"。
- **类比**：高速公路的吞吐是"每小时通过多少车"，延迟是"你这一辆从上匝道到
  下匝道用了多久"。两者经常此消彼长（压满了吞吐高但每辆车都慢）。
- **本项目**：环基准里 SPSC 测吞吐（ns/op 越小吞吐越高），ping-pong 测往返
  延迟；容量曲线里轮速率是吞吐、首响 p99 是延迟。

## §6 会话与可靠性：断了线怎么办

### 会话（Session）
- **定义**：一次完整的对话上下文（含历史、状态、流水线），**独立于网络连接
  存在**。
- **本项目的关键设计**：连接（WebSocket）会断会换，但会话不死——断线后会话
  还活着等你重连（idle 宽限期内），重连上来接着聊。连接只是会话的临时
  "附身"（attachment）。

### Epoch（纪元 / 世代号）
- **定义**：一个**只增不减**的整数，每次重连 +1，用来区分"新连接"和"旧连接"。
- **类比**：换了新手机号通知所有人——旧号码再打进来的，一律当过期处理。
- **本项目**：防"僵尸连接污染"——一个卡住的旧连接突然复活发帧时，它的
  epoch 比当前小，直接拒收。epoch 绑在连接上、不绑在帧上。

### 序列号（Sequence Number）/ 水位去重（Watermark Dedup）
- **定义**：每帧带一个递增的编号（seq）。重连后客户端可能重发一些帧，
  服务器靠"已处理到的最大编号（水位）"判断：编号 ≤ 水位的是重复，丢弃。
- **类比**：收快递时记下"已经签收到第 100 号"，再来个 95 号就知道是重复
  派送，拒收。
- **本项目**：因为 TCP 保证有序（§4），用一个"最大已处理编号"的水位就够了，
  **不需要复杂的重排窗口**——这是"比计划更简单时就删掉计划的复杂度"的例子。

### 重连续传（Reconnect Resume）
- **本项目**：断线重连时，客户端发 START（带老 session id、epoch+1、自己
  收到的最后一个下行序号），服务器认出是同一会话、接管、继续。下行序号
  **跨重连连续**（不重置），但下行音频**刻意不重传**——实时音频里迟到的
  旧帧是反特性，从"现在"继续才对。

## §7 可观测性：怎么知道系统好不好

### 指标（Metrics）/ Prometheus
- **定义**：系统运行时导出的各种数字（延迟、丢帧数、活跃会话数…），供监控
  系统抓取。Prometheus 是业界最流行的监控系统，它有一套**文本格式**约定。
- **本项目**：手写了那套 Prometheus 文本格式（~150 行，零依赖），`/metrics`
  端点吐出来。为什么手写？需要的功能极小，引官方库的依赖树不值当，而且
  "手写线格式"本就是这个项目的主题（帧协议也是手写的）。

### 计数器 / 直方图 / 仪表（Counter / Histogram / Gauge）
- **计数器（Counter）**：只增不减的累计值（如"总共丢了多少帧"）。
- **仪表（Gauge）**：可上可下的瞬时值（如"当前活跃会话数"）。
- **直方图（Histogram）**：把观测值分到一组"桶"里统计分布（如"延迟落在
  0~10ms 的有几次、10~50ms 的有几次"）——用来算分位数。
- **本项目**：延迟用直方图（好算分位），丢帧/会话数用计数器/仪表。

### 分位数 / p50 / p99（Percentile）
- **定义**：把一组数排序，p50 = 中位数（一半的请求比它快），p99 = 99% 的
  请求比它快（只有最慢的 1% 超过它）。
- **为什么看 p99 不看平均**：平均会被掩盖——99 个 10ms + 1 个 1000ms，平均
  才 20ms 看着没事，但那 1% 的用户体验极差。**p99 才暴露尾部延迟**，实时
  系统尤其在意尾部。
- **本项目**：打断 p99、首响 p99 是关键验收指标。容量曲线看的就是 p99 随
  并发怎么变。

### 延迟分解：模型固有时延 vs 内核开销
- **本项目的"两本账"**：首响延迟 = **模型固有时延**（ASR/LLM/TTS 自己花的
  时间，云模型几百 ms）+ **内核开销**（排队、调度、搬运等本项目该负责的）。
  分开记账，因为首响绝对值被模型主导，拿总数当 KPI 等于给内核贴模型的金。
  分解后内核才有自己的可验收指标——平台区内核开销 p99 实测 **5ms**。

### pprof / 性能剖析（Profiling）
- **定义**：Go 自带的性能分析工具——告诉你 CPU 时间花在哪些函数、内存在
  哪里分配。
- **类比**：给程序做体检，看哪个器官最耗能。
- **本项目**：M11.2 用 pprof 发现"CPU 几乎全花在网络系统调用和调度上，
  自有内核代码不上榜；内存分配 78% 是 WebSocket 库的读路径"——于是优化方向
  不是抠业务代码，而是减少库的分配（下行编码做到了零分配）。

### allocs/op（每次操作的分配次数）/ 零分配
- **定义**：基准测试报告里的一列——每次操作在堆上分配了几次内存。
- **为什么在意**：堆分配会触发垃圾回收（GC），GC 会带来延迟抖动。实时
  热路径上**零分配（allocs/op == 0）**意味着稳定、无 GC 毛刺。
- **本项目**：环形缓冲、缓冲池、下行帧编码都做到了 0 allocs/op，并把它
  **设成测试门禁**（`AllocsPerRun==0`，回退会让测试变红）。

## §8 浏览器端：声音怎么进出网页

### getUserMedia
- **定义**：浏览器 API，向用户申请麦克风权限并拿到音频流。
- **本项目**：`getUserMedia({echoCancellation:true})`——开启浏览器内置 AEC
  （§1），这是打断不误触发的前提。

### AudioContext / 采样率
- **定义**：浏览器处理音频的运行环境。可以指定采样率。
- **本项目**：`AudioContext({sampleRate:16000})`——直接让浏览器以 16kHz
  工作，**省掉手写重采样**（否则要把浏览器默认的 48kHz 降到 16kHz，很麻烦）。

### AudioWorklet
- **定义**：在**独立音频线程**里处理音频的机制，不占用主线程（UI 线程）。
- **为什么重要**：主线程一卡（比如页面在渲染），音频采集就会断断续续。
  AudioWorklet 跑在专用线程，采音稳定。
- **本项目**：`capture-worklet.js` 用它采集麦克风、归一成 16k PCM 帧上行。

### 播放缓冲 / 抖动缓冲（Playout Buffer / Jitter Buffer）
- **定义**：下行音频不会均匀到达（网络抖动，§4），如果来一帧放一帧就会
  忽快忽慢、爆音。播放缓冲先攒一小段（如 120ms），再按 PTS 平稳地吐给
  扬声器，抹平到达的不均匀。
- **类比**：视频"缓冲中…"先存一段再播，就是同一个道理。
- **本项目**：自适应播放缓冲——预存 120ms，**欠载（underrun，缓冲空了）**
  时垫静音并加深水位（上限 400ms），避免爆音。这是 D13"WS 上抖动 = 突发
  到达，靠播放缓冲抹平"的落地。

### Underrun（欠载）
- **定义**：播放缓冲被掏空了，没有数据可播（下一帧还没到）。
- **本项目**：欠载时垫一小段静音而非卡顿/爆音，并自适应加大缓冲深度，
  减少下次欠载。

## 附：一轮对话里这些概念怎么串起来

```
你对着麦克风说"你好"
  → getUserMedia + AudioWorklet 采成 16k/16bit/mono 的 PCM
  → 切成 20ms/640字节的帧，每帧打上 PTS 和 seq
  → 帧协议编码（24字节头 + 裸PCM），走 WebSocket 二进制上行
  → 服务器解帧 → 水位去重 → 【内联VAD算RMS能量】判断你在说话
  → 帧进入口环形缓冲（SPSC，满了drop-oldest）
  → ASR Runner 抽帧喂给 ASR，吐出 partial（实时字幕）和 final
  → final 进编排器 → 喂 LLM → LLM 流式吐 token（经有界channel，满了背压）
  → token 喂 TTS → 合成音频帧 → 进出口环 → 音频泵重戳会话时钟
  → 帧协议编码 → WebSocket 下行 → 浏览器播放缓冲攒120ms → 按PTS平稳播放
  → 你听到"你好，有什么可以帮你"

这时你突然插话"等一下"
  → VAD 在 RESPONDING_AGENT 状态检测到 speech_start（过了min-speech滤波）
  → 状态机触发 BargeIn → 取消当前轮的 context
  → LLM/TTS 的 goroutine 立即停止 → 清空出口环里没播的音频
  → 下行 BARGE_IN 控制帧 → 浏览器立即停播
  → 全程 < 200ms，因为取消走控制面、不排在堵塞的音频队列后面
  → 系统回到 LISTENING，听你的新话
```
