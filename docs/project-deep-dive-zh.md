## 0. 一句话定位与简历写法

**voicestream**：实时多模态语音 Agent 的流式内核（Go，~7300 行手写代码 +
455 行浏览器端 JS，103 个测试/基准函数，全量 `-race` 门禁）。它是夹在
麦克风/扬声器与 AI 模型之间的模型无关中间层，只负责**时序 / 背压 / 打断**；
ASR/LLM/TTS 是可插拔租户。

可直接使用的简历条目（按岗位侧重取舍）：

- 设计并实现实时语音 Agent 流式内核（Go）：WebSocket 二进制帧协议、无锁
  SPSC 环形缓冲、ASR→LLM→TTS 流水线编排与双背压、能量 VAD 打断状态机、
  断线重连会话层；浏览器端到端演示自然打断。
- 手写 Vyukov 槽序列无锁环，解决 drop-oldest 策略下经典双游标设计的撕裂读
  竞争；实测较 channel 快 1.3–1.7×、稳态零分配（`allocs/op==0` 作为 CI 门禁），
  并量化伪共享填充收益（去掉填充后反而比 channel 慢）。
- 构建负载 harness 压测真实热路径至容量拐点：~500 并发会话（单机功率受限
  环境）处 CPU 墙先至；过墙后背压按设计于入口卸载 21→38% 丢帧、出口零丢帧、
  无 OOM；打断内核取消 p99 在 3× 拐点负载下仍 ≤143ms（预算 200ms）。
- 自建可观测性：手写 Prometheus 文本格式导出（~150 行零依赖）+ SSE 逐轮
  瀑布仪表盘，将首响延迟分解为"模型固有时延 + 内核开销"两本账，平台区
  内核开销 p99 实测 5ms。

## 1. 问题定义：为什么语音 Agent 需要一个"内核"

把 ASR/LLM/TTS 三个模型 API 串起来，谁都会；难的从来不是"接通管子"：

1. **时序**：音频在两端是高频小帧（50fps/路、640B/帧），文本在中间是低频
   token；下行音频要能按 PTS 平稳播放，字幕 token 要能对齐到它对应的语音
   时刻。两种数据的节拍完全不同，却要在一条流水线里互相咬合。
2. **背压**：模型会慢、网络会抖、用户会多。任何一处变慢，内存要么有界、
   要么爆炸——而音频是只增不减的无穷流，没有背压设计就没有第二种结局。
3. **打断**：人类对话里"插话"是基本礼仪协议。用户开口到 Agent 停声超过
   200ms，对话感就碎了。打断要穿过 VAD→状态机→子链取消→在途清空→客户端
   停播整条链，且在系统过载时依然成立。

这三件事与模型智能完全正交——模型再聪明也救不了一个会 OOM、不能打断的
管道。所以本项目把模型整体外置为租户，**内核只做这三件事，并把它们各自
做成可量化验收的指标**（范围试金石：这项工作是让时序/背压/打断更好，还是
只是让 Agent 更聪明？前者做，后者插拔或推迟）。

类比：voicestream 之于语音 Agent ≈ nginx 之于 Web 应用——不写业务（模型），
只负责低延迟、带流控、可中断地搬运数据。

## 2. 总体架构：一条不对称的流水线

```
浏览器(AEC·采音worklet·自适应播放缓冲)
   ⇅ WebSocket 二进制帧（音频=裸PCM · 文本/控制=protobuf）
会话层（id·epoch·水位去重·断线续传）
   ↓ 读goroutine：解帧 → 去重 → 【内联VAD】→ 入口环(SPSC·drop-oldest)
ASR Runner（逐utterance开流） → finals(有界chan)
   → 编排器（一次一轮） → LLM →token chan(有界)→ TTS
   → 出口环(SPSC) → 音频泵(会话时钟重戳) → 写goroutine → WS
控制面：VAD事件 → 四态状态机 → BargeIn()=子链ctx取消+在途清空
```

架构的根是一个观察：**数据流频率不对称**。音频在两端高频（每路 50fps），
文本在中间低频（partial/token 每秒几次到几十次）。由此：

- **音频两端**用无锁 SPSC 环 + drop-oldest：实时音频"丢旧帧"是 feature
  （旧音频毫无价值），绝不阻塞传输 goroutine；
- **文本中段**用有界 channel + 阻塞回传：文本不可丢（丢 token = 永久损坏
  回复），channel 满就让上游停下来等。

两种机制在入口汇合成**一条背压链**：慢 TTS → token chan 满 → LLM 阻塞 →
finals chan 满 → ASR Runner 阻塞 → 停止抽干入口环 → 溢出表现为入口计数
丢帧。**内存有界不是测出来的，是构造出来的**——每个容器都有界，唯一的
泄压阀是丢最旧音频。（M10 实测印证：过拐点后入口丢帧 21→38%，出口丢帧
恒为 0，heap 在 34–171MB 波动无单调增长。）

控制面与数据面分离：VAD 事件、状态机迁移、打断取消都不在数据队列里排队
——这是"打断在过载下依然 <200ms"的结构性原因（见 §4.3）。

## 3. 关键设计决策与"为什么"

### 3.1 传输：WebSocket，而非 WebRTC / gRPC

**做了什么**：单条 WS 连接承载统一二进制帧；传输面收敛在 `transport.Conn`
四个方法的接口后面，上层零 WS 专有逻辑。

**为什么**：业界生产拓扑里，用户设备↔边缘的"最后一公里"（烂网、NAT）由
WebRTC SFU 或电话网关承担；越过边缘后的机房腿本就是 WS/gRPC 的天下
（Twilio Media Streams、OpenAI Realtime server 模式皆 WS）。本内核定位在
机房腿，demo 里浏览器直连只是临时充当边缘。更重要的是展示价值：WebRTC 把
分帧/抖动缓冲/拥塞控制全部黑盒化——用了它，"手写流式协议"就没得写了；
WS 让协议是自己的、可观测、可确定性压测。

**否决了什么**：WebRTC（业界正解但架空本项目的学习/展示目标，且 SDP/ICE/
DTLS/TURN 复杂度会吞掉硬核点的时间）；纯 UDP（够裸但传输基建工作量爆炸）；
gRPC（同走 TCP，对实时性无增益，故连"未来增强"的优先级都排在 WebTransport
datagram 之后——后者才能真正激活乱序/丢包处理）。

### 3.2 帧协议：24 字节手写定长头 + 混合 payload 编码

**做了什么**：`magic|version|type|seq|ts_us|length` 大端定长头；音频 payload
= 裸 PCM，文本/控制 payload = protobuf。

**为什么**：定长头零分配可解析、与具体传输无关（未来切字节流传输不需要
消息边界）；音频不套 protobuf 是因为 50fps 高频路径上 varint 编解码与拷贝
纯属浪费——PCM 本身就是最终格式；而文本/控制低频且结构会演化（M8 给
ControlPayload 加 session_id/epoch/last_seq 三个字段时零迁移成本），用
protobuf 把演化成本付在低频路径上。**频率不对称再次决定编码策略。**

### 3.3 无锁环：Vyukov 槽序列，而非经典双游标 —— 本项目最硬的一题

**做了什么**：每槽一个原子序列号（`slot.seq`），生产者写 tail 槽、消费者
CAS head；drop-oldest 时**生产者执行一次消费者动作**（CAS head 驱逐最旧），
随后照常入队。64 字节填充隔离 tail/head/计数器。

**为什么经典设计不行**：教科书 SPSC 环（两个游标、无槽元数据）的正确性
依赖"head 只有消费者动、tail 只有生产者动"。可 drop-oldest 策略下满环的
生产者必须推进 head——出现了**第二个 head 写者**，消费者可能读到正在被
覆盖的槽（撕裂读）。Vyukov 槽序列把"这个槽现在归谁"编码在槽本身：消费者
CAS head 成功才拥有槽，生产者驱逐也走同一个 CAS——竞争被结构消灭而非用锁
压制。副产品：出口排空（打断 flush 是第二个出队者）免费获得安全性。

**数字**：填充后 SPSC 37.6ns/op（无填充 85.8ns，2.3×差距——**且无填充版
比 channel 的 49.1ns 还慢**：无锁结构做错不如不做）；与 channel 对照
1.3–1.7× 胜，drop-oldest 速度持平但 channel 的 select 模拟非原子；全场景
`allocs/op==0` 是测试门禁不是观察值。

**否决了什么**：直接用 channel（drop-oldest 语义在并发下不成立——这不是
性能问题是正确性问题）；加互斥锁（音频路径上锁竞争与优先级反转）；MPMC
通用环（不需要，SPSC 假设让一切便宜，扇出已靠内联 VAD 塌掉，见 3.5）。

### 3.4 适配器契约：同步 `Stream(ctx, in, out)`，阻塞即背压

**做了什么**：ASR/LLM/TTS 统一为"同步函数 + 调用方所有的 channel"：
pipeline 拥有 goroutine 与 channel 生命周期；适配器只管读 in 写 out；
每次发送都被 ctx 守卫。约定：调用方关 in，适配器永不关 out。

**为什么**：把并发结构的所有权全部收进编排器，适配器作者不可能搞错
goroutine 泄漏或重复关闭；**阻塞发送本身就是背压协议**——适配器不需要
知道背压存在，写不进去就等，等不下去 ctx 会取消（所以打断永不挂起）。
mock 与真实云适配器在同一契约下互换，"模型可插拔"是类型系统保证而非口号。

**关键动作**：M4 就早接了一个真实云流式 LLM（OpenAI 兼容 SSE，DeepSeek
默认）——专治"mock 形状陷阱"（接口形状是不是真实模型塞得进去的，必须用
真家伙验证一次）。API key 只走环境变量，配置文件里没有任何密钥。

### 3.5 VAD 内联在读 goroutine，而非独立阶段

**做了什么**：能量 VAD（双门限滞回 + 最小语音时长 + hangover）直接跑在
传输读 goroutine 里，帧过 VAD 后才进入口环；VAD 产出的是**控制面事件**。

**为什么**：若 VAD 独立成 goroutine，入口需要扇出（一份给 VAD 一份给 ASR）
——SPSC 假设立刻破产，环就要升级成 MPMC 或加第二个环。内联后入口保持
严格单生产者单消费者，代价是 VAD 计算必须便宜（能量 RMS，每帧 ~微秒级，
实测读 goroutine 远未成为瓶颈）。**用计算换结构**：便宜的计算内联，贵的
结构保持简单。

### 3.6 打断：状态机 + ctx 取消，且取消不排队

**做了什么**：四态状态机（LISTENING / SPEAKING_USER / THINKING /
RESPONDING_AGENT），非法迁移拒绝并计数；`RESPONDING + speech_start` →
`BargeIn()`：取消 LLM→TTS 子链 ctx → 等子链 goroutine 全部退出 → 排空
出口环与在途 → 下行 BARGE_IN 控制帧 → 浏览器立即停播。

**为什么是状态机**：打断的边角案例靠 if-else 必然写漏——THINKING 时用户
又开口（取消尚未发声的轮）、SPEAKING_USER 时旧轮响应才到（取消迟到的陈旧
轮）、轮自然结束与打断竞速（trailing ResponseDone 必须是 no-op）。迁移表
把每个 (状态,事件) 格子变成显式决策，漏写即非法迁移计数报警。

**为什么过载下预算仍成立**：取消走控制面——latched channel（cap 1）+
ctx 树，不经过任何可能拥塞的数据队列。M10 实测：内核取消 p99 在平台区
1–2ms，3× 拐点负载下 ≤143ms，全程未破 200ms 预算；而同负载下行音频 p99
已经数秒。**"打断优先"是架构性质，不是调参结果。**

**口径诚实**：从"用户开口"到停声还要加 VAD 最小语音时长（默认 100ms 防
误触发的参数选择）——文档与指标把它和内核取消分开计量，绝不合并成一个
好看的数字。

### 3.7 编排器：一次一轮，finals 轮内不消费

**做了什么**：编排 select 在轮进行中把 finals channel 置 nil（Go 的 nil
channel 永不就绪），新 final 只能在 channel 里排队。

**为什么**：诱惑是"来了新 final 就取代旧轮"（supersede）——但这会让
finals 永远抽得动，ASR Runner 永不阻塞，**整条背压链静默断裂**，规格里
"下游慢→上游阻塞→入口丢帧"的场景永远不会发生。一行 nil-gate 保住了
全局性质。这是"局部好心破坏全局协议"的典型案例，也是面试里讲背压设计
最生动的切口。

### 3.8 会话层：连接会断，对话不能断

**做了什么**：Session 拥有 pipeline/历史/下行队列，连接只是可更换的
attachment；epoch 单调递增绑定连接（陈旧 epoch 帧拒收防僵尸连接污染）；
上行重放用**序列号水位**去重（CAS-max）；下行 seq 跨重连连续但**刻意不
重传**；IdleTimeout 兼任重连宽限期。

**为什么不做重排窗口**：单条 WS/TCP 连接内帧本就有序——做窗口是给不存在
的问题写代码，还得测它（CLAUDE.md 的 honesty 条款 D12/D13 就是为此立的）。
水位是有序流上去重的最小充分结构。**为什么不重传下行**：实时音频里迟到的
旧帧是反特性，重连后从"现在"继续才对——这一条与直觉相反，最值得在面试
里主动讲。

**验证**：8.4 长稳测试批量建/收会话后 goroutine 回基线；M10 中 5000 并发
会话过载后回落 **0 会话 / 7 goroutines**——无泄漏性质在 30× 于单测规模下
复验。

### 3.9 延迟两本账：模型的归模型，内核的归内核

**做了什么**：`TurnStats` 记录每阶段锚点时间戳；首响 = 模型固有时延
（ASR final 跨度 + LLM 首 token + TTS 首帧）+ **内核开销**（其余一切：
排队、调度、搬运）。ASR final 的时间戳在**产出处**盖戳而非编排器领取处
——排队等待计入内核账，不许赖给模型。

**为什么**：首响绝对值被模型延迟主导（云 LLM 几百 ms 起步），拿总数当
KPI 等于给内核贴模型的金。分解之后内核才有自己的可验收指标：平台区内核
开销 p99 = **5ms**（M10 实测，恒定），拐点处升至 72→436ms——增量全部
如实落在内核/排队账上，这正是容量分析需要的信号。

### 3.10 可观测性：手写 Prometheus 文本格式 + SSE 瀑布盘

**做了什么**：~150 行实现 counter/histogram/gauge-func 的文本暴露（原子
桶计数、float64-bits CAS 累加 sum）；SSE hub 非阻塞扇出逐轮记录（慢订阅
者被丢事件）；`dash.html` 五泳道瀑布图把"流水线重叠 vs 朴素串行"画在
每一轮上。

**为什么手写**：需要的表面积极小，官方库的依赖树与注册表机制远超需求；
且"手写线格式"正是本项目的题中之义（帧协议也是手写的）。**为什么观测者
绝不反压被观测者**：仪表盘卡顿不能让编排 goroutine 等待——所有 hook
非阻塞、所有 Observe 无锁，指标是背景板不是内核。会话计数器在回收时折叠
进进程累计值，翻滚下保持单调。

### 3.11 容量方法论：交付曲线而非数字（D9）

**做了什么**：负载 harness 驱动**真实热路径**（真帧穿过真传输/VAD/环/
背压），模型 mock 但注入真实时延形状（使每轮有可打断的"在飞"窗口）；
netem 等效的保序整形器扰动到达时序（在 TCP 之上诚实地只造"延迟尖峰与
突发交付"，不伪造乱序）；阶梯 ramp + 每步双视角采集（client 端到端样本
+ 服务端直方图差分求分位）。

**为什么**：单个并发数字是虚荣指标——换台机器就作废。曲线 + 拐点 +
"先撞哪面墙"的归因 + 复现命令才是工程资产。实测：平台区(50–300)→
拐点(~400–500，CPU 墙先至)→优雅降级区(750+，入口卸载、轮速率缓降、
无崩溃)。机器与电源状态如实标注（安静模式功率受限、loadgen 同机故为
**下界**）。

## 4. 核心难点：真正难的五件事

### 4.1 drop-oldest 下的无竞争驱逐（见 3.3）

教科书不会告诉你 drop-oldest 会把 SPSC 变成"1.5 个消费者"。识别出
"驱逐 = 生产者侧的一次出队"并选择槽序列结构，是本项目最高密度的一段
设计推理。

### 4.2 打断的跨组件时序闭环

打断穿过 6 个组件（VAD→状态机→编排器→子链 ctx→出口排空→下行通知→
浏览器停播），每一跳都有竞速窗口：取消与自然完成竞速、flush 与新轮首帧
竞速、BARGE_IN 控制帧与残余音频帧的下行顺序（内核保证 flush 完成后才
入队 BARGE_IN，于是客户端看到 BARGE_IN 之后的音频一定属于新轮——压测
client 的打断后重对齐正依赖这一序保证）。测量它同样难：`cancelTurn` 在
取消发起处起表，等子链全停 + 排空完成才停表，数字才配叫"取消耗时"。

### 4.3 背压链的全局性质不被局部好心破坏（见 3.7）

背压是由 5 段各自独立的代码共同构成的全局协议，任何一段"优化"掉自己的
阻塞（supersede、扩容、丢文本）都会静默杀死整条链。防御手段是把协议写进
规格场景（OpenSpec）+ 集成测试逐项断言（慢 TTS → 入口丢帧计数上升、内存
不破上限）。

### 4.4 压测的测量学：harness 自己也会骗人

M10 压测把 harness 自身压出了四个测量学 bug，每个都修复并固化为测试：

1. **瞬态 ≠ 拐点**：单窗口毛刺（GC/OS 调度）曾被误判为容量拐点——容量
   耗尽不会因加更多负载而自愈，故拐点判定要求降级**延续到下一更重的步**。
2. **相位锁定**：mock 驱动的轮时长几乎相同，同一时刻入场的 500 个 worker
   永久同步成波峰，测的不再是稳态容量——新 worker 摊开在整个 warmup 窗口。
3. **迟到响应污染**：超时轮的残余音频伪装成下一轮首响，产出"过载后 p50
   反而变好"的伪象——超时后先静默重对齐再开新轮。
4. **t0 口径吸收扰动**：首响 t0 若取整形后的实际发出时刻，netem 的 40ms
   基础延迟会被吸进 t0、在 p50 上不可见——t0 改取"用户停止说话"的理想
   网格时刻，扰动如实显形（+58ms），而服务端口径纹丝不动（内核从到达域
   计时）——两本账的分界在扰动下依然清晰。

这一节是面试的差异化素材：**会压测的人很多，知道测量系统如何撒谎的人少。**

### 4.5 浏览器端的实时音频纪律

`AudioContext({sampleRate:16000})` 让浏览器做重采样（免去手写）；
AudioWorklet 采音避开主线程卡顿；自适应播放缓冲按 PTS 吐帧、欠载时垫
静音并加深水位（上限 400ms）——WS/TCP 的"抖动"表现为突发交付，播放
缓冲是抹平它的唯一正解（D13 的诚实边界：不存在应用层乱序，所以不做
乱序处理，做了反而是谎言）。

## 6. 面试高频追问与答法

**Q：为什么不用 WebRTC？这不是语音的标准答案吗？**
A：分清两条腿。最后一公里（设备↔边缘，烂网/NAT）确实是 WebRTC 的地盘；
但越过边缘的机房腿业界本就用 WS（Twilio、OpenAI Realtime 皆是）。本内核
定位机房腿；且 WebRTC 会把分帧/抖动/拥塞全黑盒化，"手写流式协议"的学习
与展示价值就没了。演进路径留的是 WebTransport datagram（QUIC），那才会
真正激活乱序处理——gRPC 同走 TCP，对实时性无增益，优先级反而靠后。

**Q：channel 不也能做缓冲吗，为什么手写无锁环？**
A：先给数字（1.3–1.7×，双方零分配），再给真正的理由：drop-oldest 语义。
channel 的"满了挤掉最旧"只能用 select 非阻塞发→驱逐→重试模拟，与活跃
消费者并发时驱逐和重发会交错，语义不成立；环的逐槽序列号让驱逐免竞争。
然后主动补刀：文本中段我们恰恰用的是 channel——文本不可丢，阻塞回传正是
想要的背压；无锁环只在音频两端赚钱。最后给反例镇场：去掉伪共享填充的环
比 channel 还慢（85.8 vs 49.1ns）——无锁结构做错不如不做。

**Q：丢音频帧不会毁掉 ASR 吗？**
A：丢帧只发生在溢出时——那已经是异常背压状态，且丢的是最旧帧（实时音频
里价值最低）；常态下靠背压链让上游减速而不丢。丢帧率全程计数并导出指标，
压测证明过墙后丢帧率与延迟平滑抬升而非崩溃——降级是设计出来的行为。

**Q：你说打断 <200ms，这个数字怎么构成的？**
A：三段，分开计量：VAD 最小语音时长（默认 100ms，防误触发的参数选择，
可调）+ 内核取消（cancel 发起到子链全停+在途清空，p99 1–2ms，过载 ≤143ms）
+ 下行通知与客户端停播（毫秒级）。合并报一个数是不诚实的——比如把 VAD
滤波窗调成 0 能让总数变好看，但误触发会毁掉对话。

**Q：怎么保证内存有界？**
A：构造而非测试：所有容器有界（两个环、两类有界 channel、下行队列 cap
256），唯一的无穷输入（音频）在入口被 drop-oldest 斩断。测试只是复核：
soak 测试 goroutine 回基线、5000 会话过载 heap 有界且事后归零。

**Q：mock 压测能代表真实模型吗？**
A：压测度量的是内核而非模型吞吐（D9），mock 注入了真实时延形状（流式
token 节奏、合成速率）使背压与打断窗口真实存在；接口形状用真实云 LLM
（SSE）验证过。真模型接入后变化的是"模型固有时延"那本账，内核开销那
本账不变——两本账分开正是为了这一天。

**Q：瓶颈在哪？继续优化你会做什么?**
A：profile 给的答案是 syscall + 调度（CPU top-14 全是 runtime，自有代码
不上榜；分配 78% 在 WS 库读路径）。所以下一步不是抠业务代码，而是：读侧
缓冲池化（需把回收纪律穿透 adapter 契约，已写明分析留作独立 change）、
WS 库换可控读缓冲的实现或 WebTransport、多机压测剥离 loadgen 同机争用。

## 7. 工程过程本身也是亮点

- **规格驱动**：OpenSpec 管理 1 个 change、57 个任务、12 个里程碑；决策
  写进 `design.md`（D1–D13），需求写进 `specs/`（8 个能力域、每条带
  GIVEN/WHEN/THEN 场景），实现偏离规格必须改规格而非静默漂移。
- **三绿门禁**：每次提交前 `go build` / `go vet` / `go test -race` 必须
  全绿（含纯文档提交），golangci-lint 零 issue；CI 含 `allocs/op==0`
  基准门禁。
- **每里程碑一篇中文设计文档**（M2–M11，共 14 篇），记录"为什么"与
  否决的备选——本文的所有内容都可溯源。
- **诚实文化落在条款上**：不声称 TCP/WS 已提供的行为（无会话内重排窗口）、
  不重传实时音频、打断口径拆开报、容量曲线标注机器/电源状态与同机争用、
  基准给出对自己不利的发现（drop-oldest 速度持平、无填充版更慢）。
  简历上每个数字都经得起复现：命令就在文档里。

## 8. 已知局限与下一步（被问"项目还有什么不足"时的体面答案)

- 入口读侧每消息仍有 WS 库分配（io.ReadAll + 超时管道）；零分配需改
  adapter 缓冲生命周期契约——分析已写明，刻意不做一半。
- 容量数字是单机功率受限环境的下界；分机压测、性能电源模式下拐点会
  显著右移（方法与工具已就绪，曲线可一键复现）。
- VAD 是能量双门限——嘈杂环境下应换 ML VAD（接口已可插拔）；AEC 依赖
  浏览器，回声路径未自建。
- FFmpeg 编解码、WebTransport、本地模型栈（whisper.cpp/llama.cpp/Piper）、
  Redis/Kafka 外设均已拆为独立 change 列入路线图——v1 范围收敛是决策
  而非遗漏（D11）。
